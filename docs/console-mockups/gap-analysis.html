What the GUI Needs That Does NOT Exist Yet

Soup Kitchen Need Gap Effort
Centralized web server No HTTP web app exists. Health server is per-instance, minimal. No HTML/CSS/JS serving. Major — new process needed
Aggregated fleet view Each instance runs independently. No process knows about all instances at once. GUI must poll each health port or read all config.json files. Medium — discovery + aggregation
WebSocket real-time events No event streaming. Health is poll-only. No pub/sub for message events, connection state changes, agent turns. Major — new transport layer
Heartbeat history Health is current-state only. No historical uptime data stored. Need time-series for the heartbeat strips. Medium — periodic health snapshots to DB
Group concept Instances are flat (personal, q, loops, besbot). No grouping mechanism. The GUI needs a "Group" layer above instances. Small — config/metadata only
Cross-instance message inbox Each instance has its own SQLite DB. No unified view. GUI must query multiple DBs or aggregate. Medium — multi-DB reader
Activity feed / event log Logs go to pino files + journald. No structured event stream for "msg received", "reply sent", "rule triggered". Medium — event bus or log tail
Rule engine No rule/automation engine. Command router exists but is instance-internal. No keyword triggers, auto-replies, or escalation rules exposed. Major — new subsystem
Metrics aggregation No response time tracking, message volume over time, API call counts. Rate limits exist but not exposed as metrics. Medium — counters + time-series
Config editing via GUI config.json is static files on disk. No API to update instance config, change models, modify access lists at runtime. Medium — config write API + reload
Agent budget tracking tokenBudget exists in config but no real-time consumption tracking visible. agent_sessions tracks message_count but not token spend. Medium — token counter per session
QR auth flow via GUI Auth is CLI-only (bootstrap-auth.ts renders QR to terminal). No web-based QR pairing. Medium — QR to WebSocket + web render

Architecture Implication

The GUI is a new sidecar process that:

  • Discovers instances by scanning ~/.config/whatsoup/instances/*/config.json
  • Polls health from each instance's healthPort (GET /health)
  • Reads databases directly (SQLite WAL mode supports concurrent readers) from ~/.local/share/whatsoup/instances/*/bot.db
  • Calls MCP tools via each instance's Unix socket at ~/.local/state/whatsoup/instances/*/whatsoup.sock
  • Sends messages via POST /send on the health server
  • Streams events — needs a new event bus (instances emit, GUI subscribes)
  • Manages config — writes config.json + signals instance to reload
